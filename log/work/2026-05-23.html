技術深挖 (Technical Deep Dive)

1. TransportControl::record(bool) 的武裝依賴

錄音啟動的前提是目標軌道必須有 InputDeviceInstance 且 isRecordingEnabled 為 true。在模組化開發中，我們確保 MidiController 負責路由與武裝，而 RecordingController 僅負責觸發狀態切換。

2. 測試環境中的非同步初始化

在自動化測試中，ensureContextAllocated() 是非同步的。若立即檢查設備實例會得到空結果。採用「狀態機」每 500ms 檢查一次狀態，能有效解決 Headless 環境下的初始化競爭問題。

3. MIDI 注入 (Injection) 驗證

使用 engine.getDeviceManager().injectMIDIMessageToDefaultDevice() 可以在硬體層級模擬輸入事件。這是驗證錄音邏輯最完整的方式，因為它會走完從輸入設備 -> 軌道路由 -> 錄音緩衝區 -> 生成 Clip 的完整路徑。

4. 後端數據推播 (Push) vs 拉取 (Pull)

為了確保 Clip 生成後的即時性，我們選擇在 AudioEngine 層實作 Timer 定時推播。雖然增加了少量通訊開銷，但解決了錄音非同步結束後前端無法得知狀態的問題。配合 EditItemID 持久化系統，實現了音軌狀態的無縫同步。

5. 錄音流程除錯總結 (Bug Fixes)

針對用戶報告的錄音失敗與測試音問題，最終診斷結論如下：

  • 測試音來源：setupTestScene 殘留的 MIDI Clip 與全域 MonitorMode 開啟所致。修正：移除測試 Clip 並將全域 MonitorMode 設為 OFF。
  • 錄音失敗來源：在變更武裝狀態後未立即呼叫 ensureContextAllocated()，導致引擎未刷新路由。修正：在 setTrackInput 完成後立即強制刷新上下文。
  • 同步來源：錄音停止後需即時推播。修正：在 stopAndSave 後增加一次性 100ms 延遲同步。
